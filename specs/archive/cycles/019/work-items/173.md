### idx_edges_composite — notes/144.md clarification

`specs/plan/notes/144.md` specifies:
```sql
CREATE INDEX idx_edges_composite ON edges(source_id, target_id, edge_type);
```

This index was not created in the actual `schema.ts` implementation. Cycle 018 spec-adherence flagged this divergence.

**Resolution**: The UNIQUE constraint on `edges(source_id, target_id, edge_type)` (added as part of the schema design) causes SQLite to create an implicit B-tree index on those three columns, which serves the same lookup purpose as `idx_edges_composite`. The explicit index is therefore redundant.

Update `specs/plan/notes/144.md` in the Edge table section to note:
> `idx_edges_composite` was not created — the UNIQUE constraint on `(source_id, target_id, edge_type)` creates an implicit equivalent index. The explicit index definition is dropped.

# WI-167: domainQuestions.addressed_by column

## Context

`EDGE_TYPE_REGISTRY.addressed_by.source_types` in `schema.ts` includes `"domain_question"` as a valid source type for `addressed_by` edges. This means the registry expects `domain_question` YAML objects to have an `addressed_by` field that gets indexed into the `edges` table.

However, the `domainQuestions` Drizzle table in `db.ts` does not have an `addressed_by` column. The findings table does (added in WI-152). The edge-based tracking works (it goes through the `edges` table) but the column-based denormalization on `domainQuestions` is absent.

This means queries that look up `addressed_by` via the `domainQuestions` table column find nothing, even for questions that have been addressed.

## Fix: add addressed_by column

### In schema.ts

Add `addressed_by TEXT` to the `domain_questions` table DDL in `createSchema`. The column is nullable (a question may not yet be addressed).

Increment `CURRENT_SCHEMA_VERSION` from 3 to 4. The startup schema version check will delete and recreate the database on version mismatch — this is the correct behavior for the development phase.

### In db.ts

Add `addressed_by: text('addressed_by')` (nullable) to the `domainQuestions` Drizzle table definition.

### In indexer.ts (buildRow for domain_questions)

The `buildRow` function currently maps YAML document fields to database columns. Add `addressed_by` to the domain_questions mapping:

```typescript
addressed_by: toStrOrNull(doc.addressed_by),
```

Look for the existing `buildRow` mapping for domain_questions (it will have `id`, `type`, `domain_name`, `title`, etc.) and add `addressed_by` there.

### In schema.test.ts

Add a test assertion that the `domain_questions` table has an `addressed_by` column. Follow the pattern used for the existing `addressed_by` test on the `findings` table.

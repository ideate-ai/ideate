# Policies: Artifact Structure

## P-6: The artifact directory is the sole inter-phase contract — accessed exclusively through MCP tools
All inter-phase state resides in YAML files in the `.ideate/` directory. No in-memory state carries between skill invocations. Each skill starts cold and reads what it needs exclusively through the ideate MCP artifact server. Skills do not read or write YAML files directly — the MCP server is the mandatory interface between skills and the artifact directory. The SQLite index is a derived cache backing the MCP tools; skills never interact with it directly.
- **Derived from**: GP-8 (Durable Knowledge Capture)
- **Established**: planning phase
- **Amended**: cycle 016 — clarified YAML as write target, SQLite as read-only derived cache
- **Amended**: cycle 026 — MCP artifact server is now the mandatory interface; skills do not access YAML directly
- **Status**: active

## P-7: Journal entries are write-once YAML files
Each journal entry is an individual YAML file in `.ideate/cycles/{NNN}/journal/`. Once written, a journal entry file must not be edited or deleted. Chronological ordering is maintained by sequential filenames (J-{cycle}-{seq}) and timestamp fields. The journal is the authoritative chronological record of project decisions and execution history.
- **Derived from**: GP-8 (Durable Knowledge Capture)
- **Established**: planning phase
- **Amended**: cycle 016 — changed from append-only `journal.md` to individual write-once YAML files under the cycle directory structure
- **Status**: active

## P-8: Acceptance criteria use the most appropriate validation method
Work items must express acceptance criteria with an explicit validation method. For objectively measurable criteria, use machine verification (test pass/fail, type checks, structural assertions, behavioral contracts). For inherently subjective criteria (aesthetics, UX design, content tone), use human-in-the-loop verification with an identified approval authority. Subjective decisions resolved during planning become objective specs — subsequent validation checks against the documented choice, not the subjective dimension. Criteria with no stated validation method signal unresolved ambiguity in the spec.
- **Derived from**: GP-1 (Spec Sufficiency), GP-13 (Appropriate Validation Strategy), constraint C-7
- **Established**: planning phase
- **Amended**: cycle 026 — expanded from machine-verifiable-only to include human-in-the-loop validation for subjective criteria
- **Status**: active

## P-9: Work item numbers are globally unique and continue from the highest existing number in refinement cycles
New work items in a refinement cycle must be numbered starting from the highest existing work item number plus one; no two work items may share the same three-digit prefix.
- **Derived from**: GP-4 (Parallel-First Design) — execution tooling globs by prefix; duplicates cause ambiguous ordering
- **Established**: planning phase
- **Status**: active

## P-17: Every .ideate/ directory must contain a config.json with a schema_version field
The plan skill creates `.ideate/config.json` with `{"schema_version": N}` during initial directory scaffolding; the file is written once and updated only by migration scripts, never by execute, review, or refine phases.
- **Derived from**: D-17 (manifest.json as schema version marker)
- **Established**: cycle 003
- **Amended**: cycle 016 — renamed from `manifest.json` to `.ideate/config.json` as part of v3 architecture
- **Status**: active

## P-25: All machine-actionable artifacts are YAML
Every artifact that the system needs to read programmatically — work items, findings, domain policies, decisions, questions, journal entries, principles, constraints, module specs, research findings — is a YAML file in `.ideate/`. No markdown files serve as source of truth. Human-readable output (reports, summaries) is generated by code on demand, not stored as artifacts.
- **Derived from**: GP-8 (Durable Knowledge Capture); v3 architecture decision (D-006/D-010 in cycle 016 decision log)
- **Established**: cycle 016
- **Status**: active

## P-26: SQLite index is a derived cache — MCP artifact server is the mandatory access layer
`index.db` is gitignored, rebuilt deterministically from YAML files on server startup, and disposable. The MCP artifact server is the sole interface between skills/agents and the artifact data. Skills do not query SQLite directly and do not read YAML files directly — they call MCP tools. Write tools write YAML first, then synchronously update the index. Deleting `index.db` and restarting the server must produce an identical database.
- **Derived from**: GP-8 (Durable Knowledge Capture); v3 architecture decision (D-006 in cycle 016 decision log)
- **Established**: cycle 016
- **Amended**: cycle 026 — MCP artifact server elevated from optional read path to mandatory access layer
- **Status**: active

## P-27: One file per artifact
Each work item, finding, policy, decision, question, journal entry, principle, and constraint is a separate YAML file with a unique ID. No consolidated files for artifact types that multiple agents or developers might create concurrently. This prevents merge conflicts in multi-developer workflows and ensures the indexer can process individual artifacts independently.
- **Derived from**: GP-4 (Parallel-First Design); multi-developer workflow analysis in cycle 016
- **Established**: cycle 016
- **Status**: active

## P-28: Cycle-scoped directory structure for high-volume artifacts
Cycle-specific artifacts (findings, journal entries) live under `.ideate/cycles/{NNN}/`. Cross-cycle artifacts (work items, policies, decisions, principles, constraints, modules, research, interviews) live at the top level of `.ideate/`. The indexer traverses the full tree recursively — directory structure is organizational, not functional. The `type` field in each YAML file determines which SQLite table it maps to, regardless of file location.
- **Derived from**: GP-8 (Durable Knowledge Capture) — keeps directories manageable over long-lived projects with many cycles
- **Established**: cycle 016
- **Status**: active

## P-29: Graph relationships use reversed-direction scalar fields on the target artifact
The `addressed_by` field on findings and `amended_by` field on domain policies are nullable scalar fields (at most one value, not arrays). When a new work item is created to address a finding, the agent updates the finding's `addressed_by` field. When a new policy amends an old one, the agent updates the old policy's `amended_by` field. The edge type registry derives edges from these fields at index time. This design makes "which findings are still open?" and "is this policy current?" answerable by field read without graph traversal.
- **Derived from**: v3 graph architecture decisions in cycle 016; cardinality analysis showing both relationships are always scalar
- **Established**: cycle 016
- **Status**: active

## P-30: Every column used as a filter predicate in the rebuild pipeline must have a database index
Any SQLite column queried with a `WHERE` clause inside the `rebuildIndex` loop — including all file-path lookup columns on typed tables — must have a corresponding `CREATE INDEX` statement in `createSchema`. Prepared statements that are used repeatedly in a loop must be created before the loop, not inside it. Failure to index filter columns converts `O(log n)` lookups into full table scans that compound for every file processed.
- **Derived from**: D-85; GP-8 (Durable Knowledge Capture) — rebuild correctness depends on rebuild completing within acceptable time bounds as artifact counts grow
- **Established**: cycle 019
- **Status**: active

## P-31: YAML artifact IDs must use type prefixes for global uniqueness
All artifact IDs in YAML files must use a type prefix that identifies the artifact kind: WI- (work items), F- (findings), P- (policies), D- (decisions), Q- (questions), GP- (guiding principles), C- (constraints). The v1 schema's class table inheritance uses a single `nodes.id` primary key namespace across all artifact types; type prefixes prevent cross-type ID collisions by construction.
- **Derived from**: D-117; v1 class table inheritance architecture (single PK namespace)
- **Established**: cycle 022
- **Status**: active

## P-32: MCP availability checks are for external tools only — the ideate artifact server is always present
Skills use MCP availability checks (tool list inspection by name suffix) only for external/third-party MCP servers that enhance ideate's capabilities (e.g., session orchestration, remote workers). The ideate MCP artifact server is a required component of v3 and is always available — skills call its tools directly without availability checks. This enables ideate to discover and leverage external tools while maintaining a stable core interface.
- **Derived from**: GP-8 (Durable Knowledge Capture); user decision in cycle 026 refinement interview
- **Established**: cycle 026
- **Status**: active

# Decisions: Artifact Structure

## D-5: All artifact files are Markdown; no binary or structured-data formats in the artifact directory
- **Decision**: Every artifact (steering docs, plan files, reviews, journal) is a Markdown file; structure is expressed through headings, lists, and fenced code blocks, not JSON or YAML files.
- **Rationale**: Markdown files are human-readable and auditable, which the interview established as a requirement ("artifacts should be readable and auditable"); they are also directly usable as agent input without parsing.
- **Source**: specs/plan/work-items/011-artifact-conventions.md, specs/steering/interview.md (2026-03-08)
- **Status**: superseded
- **Amended**: cycle 003 — manifest.json is a deliberate exception; see D-17.
- **Superseded**: cycle 016 — v3 architecture replaces all markdown artifacts with YAML. Human-readable output is generated by code on demand. See P-25, D-58.

## D-6: Module spec layer is optional for small projects
- **Decision**: Projects with fewer than 5 logical components skip the `plan/modules/*.md` layer and decompose directly from architecture to work items; the module layer is required when components have non-trivial interfaces between them.
- **Rationale**: Constraint C-8 (Progressive Decomposition) requires the tool to detect scale and skip intermediate levels when unnecessary to avoid overhead on small projects.
- **Source**: plan/architecture.md §6 (When to Use Modules), constraint C-8
- **Status**: settled

## D-7: specs/artifact-conventions.md is plugin documentation, not a per-project artifact
- **Decision**: The canonical artifact format reference lives in the plugin's own `specs/` directory, not inside a user project's artifact directory.
- **Rationale**: Format conventions apply globally to all projects using ideate; per-project duplication would diverge; the conventions file is a plugin-level contract.
- **Assumes**: Users can locate the conventions file by examining the plugin directory.
- **Source**: specs/plan/work-items/011-artifact-conventions.md (File Scope note)
- **Status**: settled

## D-17: manifest.json is the artifact directory schema version marker
- **Decision**: Every artifact directory contains a `manifest.json` at its root with `{"schema_version": N}`. This is the sole non-Markdown file in the artifact directory, amending D-5. The manifest is informational only — no skill checks or enforces the version at runtime. Migration scripts read it; the workflow does not.
- **Rationale**: Two prior breaking schema changes required ad-hoc migration scripts. A persistent version marker lets future migration scripts detect the schema version and apply targeted upgrades without guessing directory structure.
- **Assumes**: Schema version 1 represents the current artifact directory layout; prior migrations are not retroactively numbered.
- **Source**: archive/cycles/003/decision-log.md D1, D3, D4
- **Status**: settled

## D-18: Skills are not versioned and do not enforce version compatibility
- **Decision**: Versioning applies only to the artifact directory schema. Skills do not carry version numbers and do not check `schema_version` at invocation time.
- **Rationale**: Skills do not yet need migration or quality gates; keeping versioning scope narrow reduces complexity.
- **Source**: archive/cycles/003/decision-log.md D2
- **Status**: settled

## D-21: The canonical key for artifact directory in .ideate.json is artifactDir (camelCase)
- **Decision**: The `.ideate.json` configuration file uses `artifactDir` (camelCase) as the key for the artifact directory path. Any script, skill, or documentation that reads or documents this key must use camelCase. Snake_case (`artifact_dir`) is incorrect and caused two critical bugs in the cycle 004 capstone review of report.sh.
- **Rationale**: WI-095 was created specifically to fix a snake_case/camelCase mismatch introduced in the WI-094 planning note. All skills consistently use `artifactDir`; the planning note for report.sh auto-discovery used `artifact_dir`, which was not caught during planning.
- **Source**: archive/cycles/006/decision-log.md D6; archive/cycles/006/review-manifest.md (WI-095 entry)
- **Status**: settled

Severity counts are not present at the top level of the event object.

- **Assumes**: The `by_reviewer` sub-objects include `suggestion` for schema symmetry; no current consumer parses `by_reviewer` (see Q-16).
- **Source**: archive/cycles/006/decision-log.md D3, D10; archive/cycles/006/gap-analysis.md SG1
- **Status**: settled

## D-58: All machine-actionable artifacts are YAML; markdown is superseded as source of truth
- **Decision**: v3 replaces all markdown artifacts with YAML. Review outputs become structured YAML finding objects. Domain knowledge (policies, decisions, questions) becomes individual YAML files. Journal entries become individual YAML files. Human-readable output is generated by code on demand. This supersedes D-5.
- **Rationale**: Markdown is human-readable but not machine-parseable. The v3 architecture requires all artifacts to be indexable into SQLite. Findings trapped in markdown prose cannot be queried, cross-referenced, or addressed by graph edges. YAML provides consistent programmatic access while code generates human-readable views on demand.
- **Source**: cycle 016 post-review discussion; analysis of three developer scenarios (solo, subset, late joiner) showing gaps when review outputs are markdown
- **Status**: settled

## D-59: `addressed_by` and `amended_by` as reversed-direction scalar fields on target artifacts
- **Decision**: The `addresses` and `amends` edge types use reversed-direction yaml_fields on the target artifact rather than fields on the source artifact. `addressed_by: WI-NNN` (nullable string) on findings. `amended_by: P-NNN` (nullable string) on domain policies. Both are scalar — a finding has at most one work item that addresses it; a policy has at most one successor. The edge type registry extracts these as graph edges at index time. This supersedes D-56.
- **Rationale**: Cardinality analysis: both relationships are many-to-one from the target's perspective (scalar, not array). The important queries are target-centric ("which findings are open?" = `WHERE addressed_by IS NULL`; "is this policy current?" = `WHERE amended_by IS NULL`). Reversed direction avoids array fields on work items and makes status a field read rather than a graph join.
- **Assumes**: Finding YAML files are mutable (in `.ideate/cycles/{NNN}/findings/`, not in an immutable archive). Agents creating work items update finding files atomically. Failure between writes leaves findings as `addressed_by: null` (safe — appears unaddressed, not corrupted).
- **Source**: cycle 016 post-review discussion; cardinality and query pattern analysis
- **Status**: settled

## D-60: Cycle-scoped directory structure with recursive indexer traversal
- **Decision**: `.ideate/cycles/{NNN}/` holds cycle-specific high-volume artifacts (findings, journal entries). Cross-cycle artifacts (work items, policies, decisions, principles, constraints, modules, research, interviews) live at the `.ideate/` top level. The indexer's `walkDir` traverses the full tree recursively. The `type` field in each YAML file routes it to the correct SQLite table regardless of file location. Directory structure is organizational, not functional.
- **Rationale**: Flat directories become unmanageable over many cycles (50 cycles x 20 findings = 1000 files). Cycle-scoped subdirectories keep per-cycle artifacts grouped (~20 files each). Cross-cycle artifacts are lower volume and don't need cycle scoping. The indexer already traverses recursively, so directory structure has zero code impact.
- **Source**: cycle 016 post-review discussion
- **Status**: settled

## D-61: One file per artifact for merge-conflict safety
- **Decision**: Each artifact (work item, finding, policy, decision, question, journal entry) is a separate YAML file with a unique ID. No consolidated files for artifact types that multiple agents or developers might create concurrently. The existing `work-items.yaml` consolidated format will be migrated to individual files.
- **Rationale**: Two developers creating different findings in parallel produce different files — no merge conflicts. A consolidated file (e.g., `work-items.yaml`) would conflict when two developers add items concurrently. Individual files are also more natural for the indexer (each file = one database row).
- **Source**: cycle 016 post-review discussion; multi-developer workflow analysis
- **Status**: settled

## D-62: archive/cycles/{NNN}/ markdown eliminated; replaced by .ideate/cycles/{NNN}/ YAML
- **Decision**: The `archive/cycles/{NNN}/` directory with markdown review files (summary.md, code-quality.md, spec-adherence.md, gap-analysis.md, decision-log.md) is eliminated. Review outputs become YAML finding objects in `.ideate/cycles/{NNN}/findings/`. The cycle number is a field on each finding, not a directory. "Show me cycle 16 findings" is a database query, not a directory listing.
- **Rationale**: The archive served two purposes: audit trail and quality gate memory. Both are replaced by the graph — findings with cycle numbers provide the audit trail; `addressed_by` fields provide quality gate memory. Markdown review files in `archive/` are not indexed by SQLite, creating a gap where findings are invisible to the graph across all three developer scenarios (solo, subset, late joiner).
- **Source**: cycle 016 post-review discussion; analysis of database consistency across developer scenarios
- **Status**: settled

## D-63: Drizzle ORM for type-safe query building with better-sqlite3
- **Decision**: Adopt `drizzle-orm` as the query builder for the MCP artifact server. Drizzle has first-class `better-sqlite3` support, does not require replacing the existing schema management in `schema.ts`, and eliminates the latent SQL injection risk from `upsertRow` interpolating column names into SQL strings.
- **Rationale**: The code-reviewer flagged `upsertRow` (indexer.ts:261) interpolating `Object.keys(row)` into SQL as a latent injection risk. While safe in the current call path (all rows from hardcoded `buildRow`), Phase 2 write tools will introduce external data. Drizzle prevents identifier injection by construction — typed table and column references, never raw strings. Lightweight (single `drizzle-orm` package), no dialect adapter needed.
- **Source**: cycle 016 post-review discussion; code-quality.md S3
- **Status**: settled

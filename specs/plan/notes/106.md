# WI-106: execute/SKILL.md — unverifiable self-check scrutiny

## Context

The standalone execute skill (`skills/execute/SKILL.md`) has the same worker self-check incentive problem as the brrr execute phase: workers can categorize hard-to-verify acceptance criteria as `unverifiable` rather than `unsatisfied` to avoid extra work. WI-105 adds scrutiny instructions to the incremental reviewer in `brrr/phases/execute.md`. This work item adds the same instructions to the standalone execute skill.

Note: the standalone execute skill already has the Andon cord interrupt mechanism for critical findings. No change to Andon behavior is needed here.

---

## Fix: `skills/execute/SKILL.md` — add unverifiable scrutiny to incremental reviewer prompt

Find the section in `skills/execute/SKILL.md` that describes spawning the code-reviewer after a work item completes (Phase 6, incremental review). In the reviewer's prompt, add:

```markdown
  > **Unverifiable claims**: The worker's self-check may contain criteria marked `unverifiable`. For each such claim:
  > 1. List all `unverifiable` criteria explicitly in your findings.
  > 2. Attempt to verify at least 2 of them by reading the relevant source files. If a criterion marked `unverifiable` can actually be verified by file inspection, reclassify it and report it as either `satisfied` or `unsatisfied`.
  > 3. Only accept `unverifiable` for criteria that genuinely require runtime testing, external system dependencies, or human judgment that cannot be derived from file contents.
```

---

## Acceptance Criteria

1. `skills/execute/SKILL.md` incremental reviewer prompt includes instructions to list all `unverifiable` criteria from the worker's self-check
2. The instructions require the reviewer to attempt verification of at least 2 `unverifiable` claims by reading source files
3. The instructions state that `unverifiable` is only acceptable for criteria requiring runtime testing, external dependencies, or human judgment

# WI-112: brrr/phases/review.md — add quality_summary emission after journal-keeper

## Context

`skills/review/SKILL.md` emits a `quality_summary` event to `metrics.jsonl` after each review cycle (added in WI-093). `skills/brrr/phases/review.md` is an independent reimplementation that was never updated. No quality_summary events are written for brrr-driven projects. The Quality Trends section of `scripts/report.sh` is permanently empty for brrr projects. This is Q-14 in the workflow domain, open since cycle 006.

---

## Fix: `skills/brrr/phases/review.md` — add quality_summary emission

After the journal-keeper step and before the domain-curator spawn, add:

```markdown
### Emit quality_summary Event

After journal-keeper completes, write a `quality_summary` event to `{artifact_dir}/metrics.jsonl`:

```json
{"timestamp":"<ISO8601>","skill":"brrr","phase":"6b","cycle":<N>,"event":"quality_summary","critical":<critical_count>,"significant":<significant_count>,"minor":<minor_count>,"convergence_achieved":<bool>}
```

Where:
- `critical`, `significant`, `minor` come from the findings counts synthesized from the four reviewers
- `convergence_achieved` is `false` (brrr will determine this in Phase 6c)
- `cycle` is the current 1-based cycle number

Best-effort only: if writing fails, continue.
```

---

## Acceptance Criteria

1. `skills/brrr/phases/review.md` emits a `quality_summary` event to `metrics.jsonl` after the journal-keeper step
2. The `quality_summary` event uses the same schema as `skills/review/SKILL.md`
3. The event includes last_cycle_findings (critical/significant/minor counts) and the cycle number
4. The quality_summary event is emitted before domain-curator is spawned
